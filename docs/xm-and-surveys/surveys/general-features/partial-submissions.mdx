## **Understanding Partial Submissions**

Partial submissions occur when respondents start a survey but do not complete it. These can include instances where the survey is opened but no questions are answered, or where only some questions are addressed before the survey is exited.

## **Enabling and Accessing Partial Submissions Tracking**

Tracking of partial submissions is automatically enabled for all Formbricks surveys, capturing every interaction from the moment a respondent begins the survey.

### **Types of Data Captured**

1. **Display Created**: Tracks when a survey is initially opened.

2. **Questions Partially Answered**: Records after every question & the data is handled, noting which questions were last interacted with before the survey was exited.

### **Benefits of Tracking Partial Submissions**

- **Identifies Drop-Off Points**: Pinpoints specific questions where respondents are likely to stop answering, providing critical insights into potential issues within the survey.

- **Improves Survey Design**: Data from partial submissions can guide adjustments to question complexity, survey length, or formatting to enhance respondent engagement.

- **Enhances Completion Rates**: Understanding where and why respondents are dropping off allows for targeted interventions to improve overall engagement and completion rates.

## **Analyzing Partial Submission Data**

Formbricks provides detailed analytics for partial submissions, including a per-question analysis of respondent behavior.

![Partial Submissions](/images/xm-and-surveys/surveys/general-features/partial-submissions/step-one.webp)

### **Survey Summary Analytics**

The "Analyze Drop-Offs" toggle in the survey summary reveals a comprehensive table with detailed metrics, enabling a deep dive into how respondents interact with each question:

- **Summary Metrics**:

  - **Impressions**: Total number of times the survey was viewed.

  - **Starts**: Percentage of impressions that the survey was started.

  - **Responses**: Total number of completed responses.

  - **Drop-Offs**: Percentage of starts that did not lead to a complete response.

  - **Time to Complete**: Average time taken by respondents to complete the survey.

### **Detailed Question Analysis**

Each question is analyzed for:

- **Time to Complete**: Average time taken by respondents spent on this question.

- **Impressions**: Number of times the question was viewed.

- **Drop-Offs**: Number of times respondents left the survey at this question, with a percentage indicating the drop-off rate.

This data is invaluable for pinpointing problems with specific questions and understanding the overall flow and engagement levels within the survey.

## **Use Cases**

Partial submissions tracking is particularly valuable for:

- Surveys experiencing high drop-off rates, where detailed question analysis can inform necessary adjustments.

- Studies requiring in-depth engagement metrics for each survey question to optimize content and survey structure.

## **Conclusion**

By leveraging the partial submissions tracking feature in Formbricks, you gain comprehensive insights into respondent behaviors and survey interactions. This information is vital for enhancing survey design, improving response rates, and ensuring more reliable and meaningful data collection.

## Progress Save Mechanism

Formbricks captures partial submissions on the server side via the response service. When a respondent interacts with a survey, each answer is sent to the server and persisted incrementally, ensuring that no data is lost if the respondent exits before completing the survey.

### The `finished` Flag

The core mechanism behind partial submissions is the `finished` field on the response object:

- When `finished: false` â€” the response is a **partial submission** (in-progress).
- When `finished: true` â€” the response is a **completed submission**.

The `finished` flag is set via the `updateResponse` function, which handles all response updates as the respondent progresses through the survey.

### How `updateResponse` Works

The `updateResponse` function (line 487 of `apps/web/lib/response/service.ts`) accepts the following parameters:

- **`responseId`** â€” The unique identifier of the response being updated.
- **`responseInput`** â€” An object containing `data`, `finished`, `ttc`, `language`, `variables`, and `endingId`.
- **`tx`** (optional) â€” A Prisma transaction client for ensuring data consistency across multiple database operations.

Key behaviors of the function:

1. **Data Merging**: New response data is merged with existing data using `{ ...currentResponse.data, ...responseInput.data }`. This preserves previously answered questions while adding new ones â€” ensuring incremental accumulation of responses.

2. **TTC (Time-to-Complete) Merging**: Time-to-complete metrics are preserved from previous blocks via `{ ...currentTtc, ...responseInput.ttc }`, maintaining accurate timing data across the entire survey session.

3. **TTC Total Calculation**: The total TTC is only calculated when `finished` is `true` (via `calculateTtcTotal`). This means partial responses retain per-block TTC without a misleading total value.

4. **Variable Merging**: Calculated variables are merged similarly to response data, preserving all computed values from previous blocks.

5. **Transaction Support**: The function supports an optional Prisma transaction client (`tx`), enabling atomic updates when response persistence is part of a larger database operation.

### Incremental Persistence

Response data is persisted incrementally. Each time a respondent advances to the next question or block, the current answer is sent to the server and merged into the existing response record. This batch streaming approach ensures that even if a respondent drops off at any point, all previously submitted answers are preserved.

## Resume Behavior

### Data Persistence

Each partial response is stored in the database with all previously answered questions intact. The response record accumulates data as the respondent progresses through the survey â€” each interaction adds to the existing record rather than creating a new one.

### Response Identification

Responses are tracked by a unique response ID generated when the first answer is submitted. For link surveys, the response may additionally be tied to a `singleUseId` for single-use link scenarios, ensuring that each unique link produces exactly one response record.

### Current Resume Capabilities

- Once a response is created (first answer submitted), all subsequent answers update the **same response record** within the active session.
- If a respondent closes the browser and returns via an anonymous link survey, a **new response** is typically created (no automatic cross-session resume for anonymous link surveys).
- For authenticated or identified respondents, the system can match responses via contact attributes, enabling continuation of in-progress surveys.

### Data Available in Partial Responses

Partial responses contain all data accumulated up to the drop-off point:

- All previously answered question data
- Time-to-complete metrics per question and block
- Variables calculated up to the point of disengagement
- Language selection
- Hidden field values (set at survey load time)

### Export Behavior

Partial responses appear in exports with `Finished: false` in the export data. This is included as a standard column in both CSV and XLSX exports, allowing you to filter between complete and partial responses during analysis.

<Note>
  Partial submissions are automatically included in response exports. Use the "Finished" column to filter between complete and partial responses in your analysis.
</Note>
